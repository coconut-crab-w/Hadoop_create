## Hive 的分区和分桶

### 什么是分区
> 分区是指用表的一列数值进行划分和存储的一种方式。通过分区，按照特定的维度进行组织，提高查询效率和数据管理的灵活性。

### 分区的优势

- 提高查询效率：通过分区，可以将数据按照特定的列值进行划分，使得查询只需要扫描特定的分区的数据，减少全表扫描。
- 管理数据更加灵活：可以根据业务需求对数据进行分区，方便数据的管理和维护。
- 支持数据生命周期管理：可以根据数据的时间或其他维度进行分区，方便数据的归档和清理。

### 如何创建分区表
- 在Hive中，可以使用PARTITIONED BY关键字来创建分区表。以下是创建分区表的示例：
``` sql
CREATE TABLE my_table (
  col1 INT,
  col2 STRING
)
PARTITIONED BY (dt STRING, country STRING);

```

### 如何插入分区数据
- 在向分区表中插入数据时，需要指定分区列的值。以下是向分区表插入数据的示例：
``` sql
INSERT INTO my_table PARTITION (dt='2023-01-01', country='China')
VALUES (1, 'data1'), (2, 'data2');

```
### 分区因素
在Hive中，表的分区通常基于以下几个因素：
> 时间、地理位置、类型


### HIVE的分桶

### 什么是分桶
- 分桶是将表的数据按照哈希函数计算的结果进行划分和存储。可以将数据均匀地分布到不同的桶中，提高查询的并行度和性能。

### 分桶的优势

- 提高查询性能：通过分桶，可以将数据均匀地分布到不同的桶中，使得查询可以并行地处理不同的桶，提高查询性能。

- 支持随机抽样：分桶可以方便地进行随机抽样操作，从而进行数据分析和调试。


### 如何创建分桶表
- 在Hive中，可以使用CLUSTERED BY和SORTED BY关键字来创建分桶表。以下是创建分桶表的示例：
``` sql
CREATE TABLE my_bucketed_table (
  col1 INT,
  col2 STRING
)
CLUSTERED BY (col1) INTO 4 BUCKETS
SORTED BY (col2);
```
上述示例中，my_bucketed_table表按照col1列进行分桶，分为4个桶，并按照col2列进行排序。

### 如何插入分桶数据
- 在向分桶表中插入数据时，需要使用INSERT OVERWRITE语句，并指定桶的编号。以下是向分桶表插入数据的示例：
``` sql
INSERT OVERWRITE TABLE my_bucketed_table
SELECT col1, col2
FROM my_table;

```
上述示例中，将my_table表中的数据插入到my_bucketed_table表的相应桶中。

### 分桶因素
在Hive中，表的分桶通常基于以下因素：

均匀性：为了提高查询性能，在处理大型表时可以使用哈希函数对行键值计算出一个哈希码，并将其映射到一组固定数量的存储桶中。这样可以确保相似大小和均匀性，并且在执行特定查询时可以更快地访问所需数据。



### 示例：什么是Hive支持随机抽样

- Hive分桶中的随机抽样，指的是基于分桶的物理存储结构，能够高效、均匀地从海量数据中快速抽取一小部分具有代表性的数据子集，用于数据分析、算法训练或问题调试，而无需扫描和处理全部数据。

#### 三种常见的分桶抽样方式

1. 基于桶编号的抽样（最常用,推荐的）
``` sql
-- 从总共128个桶中，随机抽取1个桶的数据（约0.78%的数据量）
SELECT * FROM user_table TABLESAMPLE(BUCKET 1 OUT OF 128 ON user_id);

-- 抽取多个桶：从每4个桶中取1个（抽取25%的数据）
SELECT * FROM user_table TABLESAMPLE(BUCKET 1 OUT OF 4 ON user_id);
```
*特点*：速度快，数据分布均匀，但抽样比例固定（1/桶总数）。


2. 基于数据比例的抽样
``` sql
-- 随机抽取0.1%的数据（百分比抽样）
SELECT * FROM user_table TABLESAMPLE(0.1 PERCENT);

-- 随机抽取1GB数据（按大小抽样）
SELECT * FROM user_table TABLESAMPLE(1G);
```
*特点*：按比例抽取，但需要扫描数据块，效率低于基于桶的抽样

3. 基于行数的抽样
``` sql 
-- 随机抽取1000行数据
SELECT * FROM user_table TABLESAMPLE(1000 ROWS);
```
*特点*：适合小数据量抽样，大数据场景效率较低。



### Flume的零点漂移问题

- 问题产生的原因：
> 1. 数据产生延迟：数据在源头产生、传输到处理系统的过程存在延迟。
> 2. 时区不一致：数据来源的时区与处理系统设定的时区基准（如UTC）不同。
> 3. 时钟不同步：数据生产端、传输中间件、处理服务器的时钟未同步。

- 解决办法：
> 1. 使用水位线（Watermark）：这是主流解决方案。系统根据已观察到的事件时间戳，推断一个延迟边界。例如，设置Watermark = 最大事件时间 - 5秒，表示系统认为比当前最大时间戳晚5秒内的数据仍可能到达，窗口会等待至此水位线后再触发计算，以此容忍一定延迟。
> 2. 在数据中携带时区信息：处理时，将数据时间戳统一转换为目标时区（如UTC）后再开窗。
> 3. 保证时钟同步：在生产端和基础设施层面，使用NTP等协议确保各环节机器时钟一致。


### 丢数据

- 场景：从kafka中获取数据到flume，再从flume中将数据传送到Hdfs中。但是在传输数据的过程中，flume无法连接Hdfs，那么在flume中的数据会消失吗？等链接hdfs成功后会少数据吗

> *核心结论*:
> 1. 数据暂存于 Channel：当 HDFS Sink 连接失败时，数据会继续堆积在 Flume 的 Channel​ 中（内存或文件）。
> 2. Sink 会重试：Flume Sink 会按照配置进行重试，直到成功或达到重试上限。
> 3. 风险点：如果 Channel 被填满，或者 Flume Agent 进程异常终止，数据可能丢失。

*整个流程是*：Kafka Source -> Flume Channel -> HDFS Sink。问题的关键在于 Channel​ 的类型和配置。

- Channel的类型决定可靠性
> - Memory Channel（内存通道）：
> 高风险：数据仅存储在内存中。
> 如果 Flume Agent 进程崩溃或机器重启，Channel 中所有未发送的数据将永久丢失。
> 等 HDFS 恢复后，只能收到新的数据，崩溃期间内存中的数据就没了。

> - File Channel / Kafka Channel（文件通道）：
> 高可靠：数据持久化在磁盘上。
> 即使 Flume Agent 进程重启，之前 Channel 中未发送的数据也能从磁盘恢复。
> 等 HDFS 恢复后，Sink 会继续从 Channel 中读取并发送数据，理论上不会少数据。

*最佳实践建议*：对于生产环境，务必使用 File Channel​ 来保证数据的可靠性。同时，合理设置 Kafka 的数据保留时间（retention.ms），为可能的数据重消费提供缓冲




### hive on spark  和  spark on hive 的区别

- 两者的核心区别在于谁主导了计算框架和SQL解析器
> 1. Hive on spark : 是以hive为主体，将最开始的MapReduce计算引擎更换成spark，但是SQL的解析和优化还是由Hive来完成。
> 2. Spark on Hive : Spark通过集成Hive Metastore来获取现有表的元数据，但SQL的解析、优化（由Catalyst优化器负责）到任务的执行，全程由Spark（Spark SQL）主导。可理解为Spark“借用”了Hive的数据目录。

现在主流生产环境的要求是“计算与存储分离”，这也符合spark on hive 的架构，只使用hive的HiveMetastore进行元数据管理，真正的计算任务交给Spark。使用spark on hive 具有性能优势Spark SQL拥有自带的Catalyst优化器和Tungsten执行引擎，其执行效率通常优于Hive的优化器转换生成的Spark任务。
